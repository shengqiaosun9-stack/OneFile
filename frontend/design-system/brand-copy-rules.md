# OnePitch Brand Copy Rules

## Brand layer
- The public-facing name is always `OnePitch · 一眼项目`.
- The product is described as a sendable project object, not as a dashboard, workspace, or editor.

## Headline rules
- Headlines must be judgment or value statements, not feature explanations.
- Headlines should describe why the object matters, not how the system works.
- Landing hero copy may mention idea, positioning, or current progress, but should always resolve into a sendable object.

## Summary rules
- `summary` always comes before `title` in visual hierarchy.
- `summary` must describe why the startup or project is worth understanding or sharing.
- `summary` must not explain OnePitch itself unless the product being described is literally OnePitch.

## CTA rules
- Buttons must use action verbs.
- Primary CTA copy should be short and imperative.
- Secondary CTAs must read like navigation or preview, not like a second conversion funnel.
- Do not use vague marketing copy like “learn more”, “discover”, or “unlock”.

## Explanation copy
- System-value language belongs in explanation blocks, helper lines, or claim/edit prompts.
- Do not let explanation copy invade the project card body.
- Explanations should be written as consequences of the user's previous action, not as abstract feature claims.

## Example card copy
- Order is fixed: summary -> title and audience -> scenario.
- Example summaries must describe the sample startup, not the card-generation system.
- Scenario copy must be concrete about who receives the object and what they understand after reading it.
