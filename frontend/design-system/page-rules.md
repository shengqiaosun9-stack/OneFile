# OnePitch Page Rules

## Landing
- Above the fold allows one object, one message, and one primary action.
- The object is Prompt Bar; the action is generate.
- The hero must not split into equal left-right halves.
- The hero must not contain a full form, login entry, project library entry, or stacked CTA rows.
- The landing hero should read as a dark brand poster with a single prompt object anchored inside it.
- Supporting explanation belongs below the fold and must read as interpretation of the previous action, not a feature list.

## Examples
- Examples are a transition layer, not a browsing surface.
- Each example must render as Example Object Card.
- Reading order is fixed: summary -> meta -> scenario -> action.
- Examples should be presented as a manual showcase carousel, not a three-card SaaS grid.
- The primary CTA sends the user back to generation; the secondary CTA only previews the object.
- Examples must not expose internal project structure, admin controls, or project-library metadata.

## `/card/[id]`
- This page is the public system surface.
- Information priority is fixed: summary -> title -> context and status -> action zone.
- The outer shell should be darker and more display-oriented than the internal edit surface.
- The page must feel sendable before it feels editable.
- Only one dominant share-oriented action may appear in the main action zone.
- It must not present itself as an archive, admin page, or settings page.

## Poster
- The poster is a propagation layer, not a page screenshot.
- Keep only the highest-signal content and a QR code.
- The poster must prioritize scanability over completeness.
- No long copy, no field matrix larger than 2x2, no settings-like labels.

## Shared page rules
- One section, one job, one primary takeaway.
- Do not use pricing-table grammar, FAQ grammar, or dashboard grammar.
- Do not let examples, project cards, and posters collapse into the same white-card layout language.
